### Review questions

1. Does the current lifespan handle both startup and shutdown explicitly?
2. Is it okay for the health check to include `environment` and `service` fields?

### Response summary

1. The current lifespan handles startup explicitly but does not handle shutdown explicitly. In `apps/api/src/rifthub_api/main.py`, the startup work happens before `yield`, and there is no cleanup logic after `yield`. If shutdown work is needed later, it should be added after `yield`, typically with `try/finally` if resource cleanup becomes important.
2. Including `service` and `environment` in the current Phase 1 health response is acceptable. `service` is low-risk and useful. `environment` is also acceptable for internal development and simple diagnostics, but it is not required for a minimal production health endpoint. If dependency checks are added later, it is usually better to distinguish basic liveness from deeper readiness or dependency health.

## 2026-03-13 20:33:00 CAT

### Review question

Why not just install `psycopg`?

### Response summary

Installing `psycopg` is a valid option, but the current Phase 2 stack is already standardized on `asyncpg` for the application runtime and the async Alembic path. The failed inspection step came from using a one-off synchronous SQLAlchemy inspection shortcut that swapped `+asyncpg` to `+psycopg`. That was tooling convenience, not a core project requirement.

Current reasoning:

- `asyncpg` is already sufficient for the FastAPI app and async SQLAlchemy engine.
- Alembic is already working against the local Postgres database without `psycopg`.
- `psql` is available for schema inspection without adding a second Python driver.
- adding `psycopg` would be reasonable only if the project intentionally wants synchronous DB tooling or sync scripts as a supported path.

## 2026-03-13 20:58:00 CAT

### Review question

`.env.example` still has everything filled out like `.env`.

### Response summary

Yes. That is currently true. It is safe only because those values are local-development placeholders rather than real secrets. But it is not ideal because it makes the example file look like the active runtime file.

The cleaner pattern is:

- `.env.example` stays committed
- `.env.example` contains only safe placeholders or obviously non-secret local defaults
- `.env` is the real local runtime file

So the current issue is more about clarity and hygiene than immediate security leakage.

### Follow-up action

The template was tightened so `.env.example` now:

- clearly states it is a template
- tells the developer to copy it to `.env`
- uses placeholder-style values instead of mirroring the active local `.env`

The docs were also updated to say runtime loads `.env`, not `.env.example`.

## 2026-03-13 21:06:00 CAT

### Review question

While reviewing `.env`, should the local Postgres user/password be changed now so it is not forgotten later in production?

### Response summary

Yes, the local password can be changed now if desired, but it should be treated as a local-development credential, not a production secret. Production credentials must be separate and managed in the deployment environment, not copied from the local `.env`.

Important nuance:

- changing `.env` alone does not automatically rotate the password inside an already-initialized Postgres container volume
- if the Docker volume was already created with the old password, either recreate the Postgres volume or change the password inside Postgres explicitly

## 2026-03-13 21:14:00 CAT

### Review questions

1. Is the fallback database URL in `packages/backend/src/rifthub_backend/config.py` risky because it can hide env-loading mistakes?
2. Could leaving that fallback in place become dangerous if it accidentally survives into production?

### Response summary

Yes on both counts.

The fallback URL is convenient for local development, but it can mask configuration errors because the app may appear correctly configured even when `.env` or deployment env vars were not loaded.

It is also a bad production posture because missing database configuration should fail fast rather than silently falling back to an easily guessable local-style connection string.

The safer pattern is:

- allow convenient defaults only for clearly local, development-only settings
- require explicit database URLs for non-local environments
- raise a configuration error when a required database env var is missing

### Follow-up action

The fallback database URL was removed.

Current behavior:

- `RIFTHUB_DATABASE_URL` is required
- missing or blank DB config raises a `ConfigError`
- `RIFTHUB_MIGRATION_DATABASE_URL` may still default from `RIFTHUB_DATABASE_URL`

This rule was also added to project memory so silent DB fallbacks are not reintroduced casually later.
